_Основная цель_: **Бесшовный переход на новую облачную архитектуру без сбоев и потери данных**  
_Потенциальная проблема_: **Проблемы при миграции в облако — человеческий фактор, отсутствие опыта**  
_Основные категории_: Процессы, Персонал, Инфраструктура, Интеграция, Программное обеспечение

---

## Анализ выявленных проблем

| **Категория** | **Проблема** | **Описание / Причина** | **Риск** |
|----------------|--------------|--------------------------|-----------|
| **Процессы** | Сложности в настройке data-pipeline | Большое количество ручных шагов, недостаток автоматизации и документации, недостаток опыта | Замедление запуска и неустойчивость пайплайнов, потеря данных при переносе |
|  | Необходимость выстраивания новой культуры работы с данными | Переход на cloud-first подход требует обучения и адаптации персонала | Сопротивление изменениям, ошибки при выстраивании новых процессов |
|  | Необходимость обучающих процессов в рамках облака | Недостаточная подготовка к использованию новых инструментов и сервисов | Ошибки конфигурации и неоптимальное использование инструментов |
|  | Изменение модели затрат (капитальные -> операционные) | Новая финансовая политика влияет на отделы и сотрудников | Риски конфликтов, финансового несоответствия на период адаптации |
| **Персонал** | Выгорание в период миграции и освоении новых технологий | Перегрузка сотрудников, необходимость быстро осваивать новые инструменты | Снижение качества работы, рост ошибок, потенциальные увольнения |
|  | Конфликты при перераспределении нагрузки и ответственности | Неявные границы ответственности, небольшое количество специалистов | Потери в коммуникации, задержки в задачах, нехватка времени на документацию |
|  | Сопротивление нововведениям и стресс | Недостаточная мотивация, страх перед изменениями | Замедление внедрения решений, конфликты |
|  | Просьбы повышения в критический момент | Эскалации по зарплате и нагрузке во время перехода | Увеличение текучести, стресс-коммуникация |
|  | Заболевание или уход ключевых сотрудников | Потеря носителей знаний в процессе миграции | Потеря критических компетенций |
|  | Нехватка компетенций для перехода в облако | Сотрудники не владеют технологиями cloud-native | Ошибки в настройке, эксплуатации, срывы сроков |
| **Инфраструктура** | Неверная настройка CI/CD, сети | Отсутствие  выстроенных регламентов для новых пайплайнов, отсутствие опыта | Сбои в деплоях, ошибки версионирования |
|  | Использование неподходящих инструментов | Например, VM вместо Optimized Image, неверно подобранный tier сервиса, использование not managed сервисов | Потеря эффективности, рост затрат, ошибки, срыв сроков |
|  | Недостаточная прозрачность после перехода в cloud | Ошибки мониторинга и логирования, так как нет выстроенной культуры соблюдения прозрачности | Отсутствие контроля SLA, потеря данных, медленная реакция на ошибки |
|  | Сложности с расчётом ценовой политики облака | Ошибки в прогнозировании и бюджетировании из-за сложности в расчете потребления квот сервисами | Превышение бюджета |
|  | Потенциальные проблемы автоскейлинга | Ошибки конфигурации балансировщика, container optimized image, statefull-компоненты, лимитов ресурсов | Сбои при пиковых нагрузках |
| **Интеграция** | Ошибки при настройке сетевых политик и доступов | Некорректная стыковка внутренних и внешних сетей | Потеря связи между системами |
|  | Неучтённые сетевые издержки (gost-gateway, VPN) | Повышенные задержки, timeouts, нестабильность | Увеличение latency, ошибки интеграции |
|  | Пробелы в настройке безопасности при интеграциях | Отсутствие end-to-end TLS и ACL | Утечки данных, нарушение ФЗ-152 |
| **Программное обеспечение** | Неверная конфигурация программных компонентов | Отсутствие стандартов настройки и версионирования | Ошибки в работе сервисов |
|  | Баги в скриптах и коде миграции | Низкое покрытие тестами и неаккуратное копирование логики | Нарушение целостности данных |
|  | Нестабильная работа компонентов после переноса | Несовместимость библиотек, различия окружений | Частичные отказы и деградация |
|  | Использование избыточных продуктов | Избыточная инфраструктура, дублирование сервисов, перекладывание отвественности на сторонний компонент, когда мы можем решить проблему имеющимися средствами | Рост затрат, рост когнитивной нагрузки на персонал, сложность системы |

## Рекомендации по устранению узких мест

| **Категория** | **Рекомендация** | **Приоритет** | **Ожидаемый эффект** |
|----------------|------------------|----------------|------------------------|
| **Процессы** | Создать стандартизированные data-pipeline с описанием DAG, DAG-триггеров и шагов, привлечь хорошего python-специалиста с опытом работы с airflow, резервные потоки данных на первое время| Высокий | Сокращение ошибок, наличие запасного (старого и временного) варианта работы с данными |
|  | Провести внутренние разборы облачной инфраструктуры, основных компонентов безопасности и важности соблюдения регламентов | Средний | Повышение компетенций и уверенности персонала |
| **Персонал** | Проводить встречи для шеринга опыта между всеми техническими специалистами вне зависимости от роли для смягчения потенциального bus-factor, подготовить денежный резерв на консультационные услуги | Высокий | Снижение выгорания, сохранение компетенций |
|  | Ввести политику bus-factor reduction для технического персонала (благо их немного) | Средний | Минимизация зависимости от ключевых сотрудников |
|  | Проводить регулярные one-to-one встречи по адаптации к изменениям | Низкий | Улучшение психологического климата |
| **Инфраструктура** | Провести аудит CI/CD, унифицировать шаблоны пайплайнов, использовать managed компоненты облака | Высокий | Повышение надёжности и предсказуемости релизов |
|  | Настроить Cloud Monitoring и Billing Alerts | Средний | Контроль бюджета и SLA |
|  | Тестировать автоскейлинг под нагрузкой (stress-тесты) | Высокий | Предотвращение отказов при пиках |
| **Интеграция** | Настроить observability для сетевых интеграций (latency, errors) | Высокий | Быстрое обнаружение проблем |
|  | Пересмотреть сетевые политики и ACL (zero-trust) | Средний | Повышение безопасности интеграций |
|  | Разработать шаблоны интеграции с учётом gost-gateway | Низкий | Предсказуемость поведения соединений |
| **Программное обеспечение** | Провести ревизию конфигураций и зависимостей, совместный разбор проектировочного решения, оценка риска (с тех.специалистами), консультации со специалистами облака, ревью референсных архитектур | Высокий | Устранение несовместимостей, формирование риск-регистров и планов реагирования |
|  | Ввести тестирование миграционных скриптов (unit+integration) | Высокий | Снижение рисков потери данных |
|  | Централизовать управление версиями ПО и библиотек | Средний | Повышение стабильности работы сервисов |

Анализ диаграммы Исикавы показал, что:
- Главные риски — **организационные (персонал, процессы)** и **инфраструктурные (настройка CI/CD и автоскейлинг)**.  
- **Ключевой приоритет** — поддержка сотрудников и прозрачность процесса + parallel run
- **Интеграционные и сетевые аспекты** требуют усиленного контроля, мониторинга из-за появления новых каналов (VPN, gost-gateway), необходимо заложить средства для консультаций по необходимости
- **Постоянный мониторинг и стресс-тестирование** помогут минимизировать сбои и снизить человеческий фактор.

Реализация этих мер обеспечит **устойчивый и предсказуемый переход** на облачную архитектуру с сохранением стабильности бизнеса.
